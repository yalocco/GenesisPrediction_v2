# Working Agreement (GenesisPrediction v2)

Version: 1.1
Last Updated: 2026-02-27

このドキュメントは「GenesisPrediction v2」を安全・確実に進めるための共同作業ルールです。
目的は **人間の事故（貼り間違え / 途中生成のコピペ / 手順抜け / 破壊的変更）を最小化** しつつ、
前進速度を落とさないことです。

---

# 1. 絶対ルール（破ると事故る）

## 1.1 1ターン = 1作業
- 1回のやり取りでは **1つの作業** だけ扱う
- 例：UI修正 / スクリプト修正 / Runbook更新 / デバッグ手順確定 など
- 併行して話し始めたら、必ず「どれを先にやるか」を決める

## 1.2 差分追加（部分修正・パッチ方式）の提案は禁止
- **必ず完全ファイル全文** で提示する
- ユーザー側は「ファイルを丸ごと置き換える」運用を基本とする
- 例外（緊急対応など）は必ず理由を明記してから

## 1.3 変更が出たら git diff → commit
- まず `git diff` を確認してから `git add` / `git commit` / `git push`
- 「試しに当ててみた」変更は、原則コミットしない

## 1.4 GUIは read-only（SST）
- GUI（`app/static/*.html`）は **JSON/CSV/PNG を読むだけ**
- GUI側で再計算・生成しない（再現性と事故防止）

# 2. コラボ手順（安全に進める）

## 2.1 ルート・パスの前提
- Repo Root: `D:\AI\Projects\GenesisPrediction_v2`
- GUI: `app/static/*.html`
- 分析成果（SST）: `data/world_politics/analysis/`
- digest: `data/digest/`（必要に応じて alias/コピー）

## 2.2 作業の基本フロー
1) 状況確認（ログ/スクショ/コマンド出力）
2) 変更対象ファイルを確定
3) **完全ファイル**を生成して上書き
4) `git diff` で意図通りか確認
5) `git add` → `git commit` → `git push`

## 2.3 “安全第一”の判断基準
- 迷ったら：
  - 破壊的変更を避ける
  - 既存安定版の動作を崩さない
  - まず観測（ログ/差分）を増やす
- “やっつけ”をすると後で必ず負債になる

## 2.4 dataはGit管理しない
- `data/` 配下は基本Git管理しない（サイズ・更新頻度・環境差）
- 例外（スキーマ/サンプル/小さな固定ファイル）は、必要性を説明してから

## 2.5 長大ファイルはダウンロードリンク方式（コピペ事故防止）

- 対象：概ね **300行以上** の HTML / CSS / JS / PowerShell / Python（特に `app/static/*.html`）
- 目的：**貼り間違え**・**途中生成のコピペ**・**行欠け** を防ぐ（人間ミス削減）
- 原則：
  - 私（ChatGPT）が **完全ファイル** を生成し、**ダウンロードリンク（sandboxファイル）** を提示する
  - あなたはそのファイルを **ダウンロードして指定パスへ上書き保存** する
  - ブラウザ上のコードをコピペして貼る運用は **原則しない**（緊急時のみ）

### 手順（推奨）

1) こちらが `sandbox:/mnt/data/<file>` のリンクを提示  
2) あなたがダウンロード  
3) 上書き保存（例：`app/static/sentiment.html`）  
4) `git diff` → `git add` → `git commit` → `git push`

# 3. 具体ルール（よくある事故を潰す）

## 3.1 “日付（as_of）”の扱い
- “今日の日付”が表示されない場合は、まず **source JSON が古い** 可能性を疑う
- GUIは “SSTの最新” を見せるだけ（GUIで無理やり直さない）

## 3.2 画像/サムネのフォールバック
- 画像が 403/404 などで取れないのは普通に起こる
- その場合は favicon などに安全にフォールバックする

## 3.3 新規スレ（新スレ移動）の基準
- 画面スクショが増え、ログが長くなり、判断が難しくなったら新スレ
- 「目的」「現状」「確定情報」「次の一手」を最初に貼ってから始める

# 4. コミットメッセージの指針
- 何を / なぜ を短く
- 例：
  - `Sentiment Trend: colorize risk/positive/uncertainty lines`
  - `Index/Overlay: prefer daily_summary as_of when view_model is stale`
