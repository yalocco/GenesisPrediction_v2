# GenesisPrediction UI Pages Reference

Purpose  
This document records the design intent, data sources, and responsibilities of each UI page in GenesisPrediction.

The goal is to ensure that future developers, AI systems, and maintainers understand:

- Why each page exists
- What data it consumes
- What responsibility it has in the architecture
- What must NOT be changed

This document acts as a **design tablet ("stone tablet")** for the UI layer.

GenesisPrediction follows a strict architecture:

Observation
↓
Trend
↓
Signal
↓
Scenario
↓
Prediction

The UI must **display results only** and must **never perform calculations**.

Calculation and decision logic belong strictly to:

```

analysis/
scripts/

```

The UI layer is **presentation only**.

---

# UI Architecture Overview

GenesisPrediction Dashboard consists of the following pages:

```

index
overlay
sentiment
digest
prediction
prediction_history

```

These pages together form the **GenesisPrediction Dashboard System**.

Each page corresponds to a different layer of the system.

| Page | Layer | Purpose |
|-----|------|------|
| index | System status | Overview of system health and latest results |
| overlay | FX + macro signals | Remittance decision and economic signals |
| sentiment | Observation | News sentiment analysis |
| digest | Observation summary | Curated daily world events |
| prediction | Prediction | Current forecast summary |
| prediction_history | Historical prediction | Track past predictions |

---

# 1. index Page

File

```

app/static/index.html

```

Purpose

System overview page.

This page provides a **quick snapshot of GenesisPrediction status**.

It is intended to answer the question:

> Is the system running and what is the current global state?

Data Sources

```

analysis/world_politics/analysis/view_model_latest.json
analysis/health/health_latest.json
analysis/world_politics/analysis/sentiment_latest.json
analysis/world_politics/analysis/scenario_latest.json

```

Displayed Components

- System health
- Global risk level
- Latest analysis timestamp
- Quick navigation to major pages

Important Notes

Index page should remain **lightweight and fast**.

It should not attempt to display detailed data.

Its role is **system dashboard entry point**.

---

# 2. overlay Page

File

```

app/static/overlay.html

```

Purpose

Overlay page visualizes the **interaction between FX markets and global macro signals**.

This page is critical for **remittance decision support**.

It answers the question:

> Is it a good time to send money internationally?

Primary Components

```

Global Status
FX Decision
FX Overlay Chart
Economic Signals

```

Data Sources

FX Data

```

analysis/fx/fx_decision_latest.json
analysis/fx/fx_overlay_latest_*.png
analysis/fx/dashboard_rates_latest.csv

```

World Signals

```

analysis/world_politics/analysis/view_model_latest.json
analysis/world_politics/analysis/daily_summary_latest.json

```

Overlay Charts

Charts are generated by:

```

scripts/fx_multi_overlay_from_rates.py
scripts/build_fx_overlay_variants.py

```

These scripts produce images such as:

```

fx_overlay_latest_jpythb.png
fx_overlay_latest_usdjpy.png
fx_overlay_latest_usdthb.png

```

Overlay page **displays images only**.

It does NOT compute anything.

Decision Priority

Pair-specific decisions override global decisions.

Example

```

USDTHB pair decision
↓
if missing
↓
global FX decision

```

Important Notes

Overlay is one of the most important pages in the system.

It combines:

- FX market signals
- macroeconomic signals
- geopolitical developments

Future extensions may integrate:

- oil price shocks
- geopolitical escalation signals
- currency volatility regime detection

# 3. sentiment Page

File

```

app/static/sentiment.html

```

Purpose

Sentiment page visualizes **news sentiment analysis**.

This page belongs to the **Observation Layer**.

It answers the question:

> What is the emotional tone of the global news environment?

Pipeline

```

news fetch
↓
news normalization
↓
sentiment scoring
↓
sentiment_latest.json

```

Data Source

```

analysis/world_politics/analysis/sentiment_latest.json

```

Displayed Elements

- Positive sentiment count
- Neutral sentiment count
- Negative sentiment count
- Sentiment balance indicator
- News articles

Important Notes

Sentiment page is **not forecasting**.

It simply reflects **the emotional environment of the news cycle**.

Prediction layers consume sentiment data indirectly through signals.

---

File

app/static/digest.html

This page transforms raw news into structured highlights.

Pipeline

```

news fetch
↓
article filtering
↓
highlight selection
↓
daily_summary_latest.json

```

Data Source

```

analysis/world_politics/analysis/daily_summary_latest.json

```

- macroeconomic events

File

```

app/static/prediction.html

Prediction is the final output of the architecture:

```

Observation
↓
Trend
↓
Signal
↓
Scenario
↓
Prediction

```

```

analysis/world_politics/analysis/scenario_latest.json

# 6. prediction_history Page

File

```

app/static/prediction_history.html

```

Purpose

Prediction history tracks how predictions evolved over time.

This page enables:

- historical verification
- model improvement
- scenario comparison

Data Source

```

analysis/world_politics/analysis/prediction_history.json

```

Displayed Elements

- previous predictions
- scenario outcomes
- confidence changes

Important Notes

Prediction history is essential for **model credibility**.

# UI Design Principles

GenesisPrediction UI follows strict rules.

Rule 1

UI must **never compute results**.

Rule 2

UI must only **display artifacts generated by scripts**.

Rule 3

Artifacts originate from:

```

analysis/
scripts/

```

Rule 4

The UI must remain **stable even if analysis evolves**.

Rule 5

Data contracts must remain stable.

Example

```

sentiment_latest.json
view_model_latest.json
scenario_latest.json

```

# Architectural Philosophy

GenesisPrediction was built under a guiding philosophy:

Technology should serve truth and understanding.

The system must remain transparent and auditable.

Future maintainers must understand:

- why decisions are made
- where data originates
- how signals propagate through the system

This document exists to prevent the system from losing its design integrity.

As a reminder of humility:

Even powerful analytical systems can drift if their foundations are forgotten.

Therefore we record the design clearly.

Not to worship the system itself,

but to remember the principles that built it.
