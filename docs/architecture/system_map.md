# GenesisPrediction System Map

Purpose  
This document explains the **directory structure and responsibilities** of the GenesisPrediction repository.

It exists so that future maintainers, AI systems, and developers can quickly understand:

- where each subsystem lives
- what each folder is responsible for
- where computations happen
- where artifacts are generated
- what the UI is allowed to access

This document acts as the **navigation map of the GenesisPrediction codebase**.

---

# Repository Overview

GenesisPrediction is organized into several major domains.

```

app/
analysis/
scripts/
data/
docs/

```


Each domain has a clearly defined responsibility.

| Directory | Role |
|--------|------|
| app | UI and dashboard pages |
| analysis | computed artifacts and results |
| scripts | computation engines |
| data | raw and intermediate data |
| docs | architecture and documentation |

---

# app/

```

app/
└─ static/

```

Purpose

User interface layer.

Location of all dashboard pages and UI assets.

Example pages

```

index.html
overlay.html
sentiment.html
digest.html
prediction.html
prediction_history.html

```

Rules

- UI must **not compute data**
- UI must **not write artifacts**
- UI must only **display artifacts**

The UI reads data from:

```

analysis/
data/world_politics/analysis/

```

---

# analysis/

```

analysis/
├─ fx/
├─ world_politics/
└─ health/

```

Purpose

Stores **analysis artifacts** generated by scripts.

Artifacts represent the **final outputs of the computation layer**.

Examples

```

sentiment_latest.json
daily_summary_latest.json
view_model_latest.json
scenario_latest.json
prediction_latest.json

```

These artifacts are read by the UI.

The analysis directory acts as the **Single Source of Truth (SST)** for computed results.

# scripts/

```

scripts/

```

Purpose

All **computation engines and pipelines** live here.

Scripts perform:

- data ingestion
- analysis
- signal generation
- scenario generation
- artifact creation

Examples

```

run_morning_ritual.ps1
build_daily_sentiment.py
build_world_view_model_latest.py
scenario_engine.py
prediction_engine.py
fx_multi_overlay_from_rates.py
build_fx_overlay_variants.py

```

Important rule

```

scripts generate artifacts
analysis stores artifacts
UI displays artifacts

```

# Execution Flow

Daily pipeline execution typically follows this sequence.

```

run_morning_ritual.ps1
↓
fetch data
↓
update observation artifacts
↓
generate signals
↓
generate scenarios
↓
generate predictions
↓
refresh analysis artifacts
↓
UI reads latest artifacts

```

This ensures the dashboard always reflects the **latest system state**.

---

# Artifact Flow

The most important concept in GenesisPrediction is **artifact flow**.

```

scripts
↓
analysis artifacts
↓
UI display

```

Artifacts act as **contracts between system layers**.

Examples

```

sentiment_latest.json
scenario_latest.json
prediction_latest.json

```

These files ensure the UI and computation layer remain loosely coupled.

---

# Architectural Principles

GenesisPrediction follows several architectural rules.

Rule 1

Separation of computation and presentation.

Rule 2

Artifacts serve as stable contracts between layers.

Rule 3

Scripts must be deterministic and reproducible.

Rule 4

UI must remain simple and stable.

Rule 5

Documentation must evolve with the system.
